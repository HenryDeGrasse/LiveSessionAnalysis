## 2. High-Risk Area Audit

### Area 1: Reconnect / Disconnect Semantics

**Verdict: Improved, but still not fully polished**

**What works now:**
- `_grace_period_finalize()` in `ws.py` is real and tested (async test in `test_reconnect.py`)
- Grace period is configurable via `reconnect_grace_seconds` (default: 10s)
- Reconnect cancels pending grace task via `cancel_grace_task()`
- Backend rejects reconnects to already-ended sessions (code 4003)
- Frontend `useWebSocket.ts` treats 4003 as terminal and does not reconnect to ended sessions
- Student disconnect now sends `participant_disconnected` instead of `session_end` during grace
- `_finalize_session()` is now responsible for sending `session_end`

**Remaining limitations:**
- The backend/frontend now emit and handle `participant_reconnected`, which clears the tutor disconnect banner.
- There is still no true browser-level reconnect smoke test, so the UX is improved and backend-websocket-tested, but not fully browser-E2E validated.

**Not fully tested:**
- No browser E2E reconnect flow
- No test proving metrics resume correctly after reconnect in a realistic media-streaming scenario

### Area 2: Tutor Identity / Session Metadata

**Verdict: Improved, but still only partially end-to-end**

**What works now:**
- `POST /api/sessions` accepts optional `SessionCreateRequest` body with `tutor_id` and `session_type`
- The frontend now sends `tutor_id` from the home page when creating a session
- `tutor_id` flows through: `session_manager.create_session()` ‚Üí `SessionRoom.tutor_id` ‚Üí `_save_session()` ‚Üí `generate_summary()` ‚Üí `SessionSummary.tutor_id` ‚Üí stored JSON
- Trends API (`/api/analytics/trends`) accepts `tutor_id` query param and filters by it

**What is still incomplete:**
- The default frontend flow now sends `session_type`, but there is still no automated frontend/browser test covering that path
- Session type is now persisted through summary generation, but the new path is only backend-tested rather than browser-E2E-tested
- This means tutor identity and session type are now meaningfully wired, but still under-validated from the real UI path

### Area 3: Adaptive FPS

**Verdict: End-to-end wired, but frontend implementation is fragile**

**What works (backend):**
- `target_fps` is populated in `MetricsSnapshot` from `room.current_fps` (engine.py line 88, ws.py line 259)
- Backend sends `target_fps` in every metrics JSON message to the tutor
- Backend rate-limits video frame processing via `should_process_video_frame()` using `current_fps`

**What works (frontend):**
- `types.ts` has `target_fps: number` on `MetricsSnapshot`
- Session page reads `metrics.target_fps` in `onMessage` and updates `targetFpsRef`
- Frame send interval is recalculated: `Math.round(1000 / targetFpsRef.current)`

**What is fragile:**
- The FPS change mechanism works by clearing the interval in `onMessage` and relying on a 500ms polling `setInterval` to notice the cleared interval and restart it (lines 158-164 of `session/[id]/page.tsx`). This creates a gap of up to 500ms where no frames are sent.
- The polling pattern is a workaround for React's inability to directly restart an interval from inside a callback without causing render loops. It works but is inelegant.
- **No frontend tests exist for this behavior**

### Area 4: Energy Baseline vs Coaching

**Verdict: Now partially baseline-aware**

**What exists now:**
- `EnergyTracker` has `_score_history`, `baseline`, and `drop_from_baseline`
- Baseline stabilizes after 10+ samples
- `drop_from_baseline` is now exposed through `ParticipantMetrics.energy_drop_from_baseline`
- `_energy_drop_condition` now fires on either:
  - absolute low energy (`< settings.energy_drop_threshold`), or
  - significant baseline drop (using the configured baseline-drop threshold)

**What is still weak:**
- The new threshold is configurable now, but there is still no strong end-to-end validation showing the coaching behavior is well-calibrated against realistic sessions
- This is a real improvement, but not yet a thoroughly validated coaching signal

### Area 5: Retention Cleanup

**Verdict: Improved, but only enforced at startup**

**What exists now:**
- `SessionStore.cleanup_expired()` method exists
- Tests confirm it deletes old sessions, keeps recent ones, and handles malformed JSON
- `main.py` now calls `cleanup_expired()` in the application lifespan startup hook

**What is still limited:**
- Cleanup runs on server start, not continuously
- There is still no periodic/background cleanup path during long-running uptime
- There is still no API/admin endpoint to trigger cleanup on demand

### Area 6: Latency Claims

**Verdict: Latency tests pass, but do NOT exercise a detected-face path**

**Evidence:**
- `test_face.jpg` fixture is a 320x240 image. Running MediaPipe FaceMesh on it returns `Face detected: False` ‚Äî confirmed via direct test
- `_make_face_jpeg()` in `test_pipeline_latency.py` draws a crude ellipse/circles ‚Äî MediaPipe also does not detect this as a face
- This means all latency tests exercise the **no-face-detected path**: gaze is `None`, expression is `None`, stage timings for gaze/expression are near-zero
- The "under 500ms" claim is technically true but **meaningless for the real hot path** where FaceMesh actually detects a face and gaze/expression stages do real work
- With a real face, FaceMesh + iris landmark extraction + gaze angle computation would add significant time; this is completely unmeasured
- **The latency tests provide false confidence**

### Area 7: Frontend Quality Gates

**Verdict: No frontend quality infrastructure exists**

- **Zero frontend test files** ‚Äî no Jest, Vitest, React Testing Library, or Playwright configured
- **No ESLint config** ‚Äî `npx next lint` prompts for setup, meaning it has never been configured
- **TypeScript passes** (verified) ‚Äî this is the only frontend quality gate
- No smoke tests for session page state transitions (consent ‚Üí media ‚Üí connected ‚Üí metrics ‚Üí session_end)
- No test for reconnect UX behavior
- No test for adaptive FPS behavior

## 7. Fixes Applied (Same Session)

- **`types.ts`**: Added `participant_disconnected` and `participant_reconnected` message types
